# Schema do Supabase para o Portal de Links

Este arquivo resume as tabelas usadas pelo backend em [backend/database.py](backend/database.py) e o que precisa existir no Supabase antes de apontar o projeto para produção na Vercel.

## Visao geral

O banco tem 4 tabelas principais:

- `empresas`: cadastro das empresas atendidas.
- `links`: links cadastrados por empresa.
- `chamados`: tickets/solicitacoes por empresa.
- `agenda_eventos`: eventos da agenda interna.

Os relacionamentos sao:

- `links.empresa_id` referencia `empresas.id`.
- `chamados.empresa_id` referencia `empresas.id`.

## Tabelas e colunas

### `empresas`

- `id` integer, chave primaria, auto increment.
- `nome` varchar(255), nao nulo, unico.

### `links`

- `id` integer, chave primaria, auto increment.
- `empresa_id` integer, nao nulo, chave estrangeira para `empresas.id`.
- `titulo` varchar(255), nao nulo.
- `url` varchar(500), nao nulo.
- `descricao` text, opcional.
- `atualizado_em` timestamp, opcional, padrao `now()`.

### `chamados`

- `id` integer, chave primaria, auto increment.
- `empresa_id` integer, nao nulo, chave estrangeira para `empresas.id`.
- `titulo` varchar(255), nao nulo.
- `descricao` text, nao nulo.
- `status` varchar(50), padrao `Aberto`.
- `observacao_ti` text, opcional.
- `criado_em` timestamp, padrao `now()`.
- `atualizado_em` timestamp, padrao `now()`.

### `agenda_eventos`

- `id` integer, chave primaria, auto increment.
- `titulo` varchar(255), nao nulo.
- `data_inicio` timestamp, nao nulo.
- `data_fim` timestamp, nao nulo.
- `local` varchar(255), opcional.
- `descricao` text, opcional.
- `criado_em` timestamp, padrao `now()`.
- `atualizado_em` timestamp, padrao `now()`.

## SQL pronto para o Supabase

Execute este bloco no SQL Editor do Supabase:

```sql
create table if not exists public.empresas (
  id bigint generated by default as identity primary key,
  nome varchar(255) not null unique
);

create table if not exists public.links (
  id bigint generated by default as identity primary key,
  empresa_id bigint not null references public.empresas (id) on delete cascade,
  titulo varchar(255) not null,
  url varchar(500) not null,
  descricao text null,
  atualizado_em timestamp without time zone default now()
);

create table if not exists public.chamados (
  id bigint generated by default as identity primary key,
  empresa_id bigint not null references public.empresas (id) on delete cascade,
  titulo varchar(255) not null,
  descricao text not null,
  status varchar(50) default 'Aberto',
  observacao_ti text null,
  criado_em timestamp without time zone default now(),
  atualizado_em timestamp without time zone default now(),
  created_by uuid null references auth.users(id)
);

create table if not exists public.agenda_eventos (
  id bigint generated by default as identity primary key,
  titulo varchar(255) not null,
  data_inicio timestamp without time zone not null,
  data_fim timestamp without time zone not null,
  local varchar(255) null,
  descricao text null,
  criado_em timestamp without time zone default now(),
  atualizado_em timestamp without time zone default now()
);

create index if not exists idx_links_empresa_id on public.links (empresa_id);
create index if not exists idx_chamados_empresa_id on public.chamados (empresa_id);
create index if not exists idx_agenda_eventos_data_inicio on public.agenda_eventos (data_inicio);

-- Tabela para armazenar senhas (modo simples). Considere criptografia ou RLS em produção.
create table if not exists public.senhas (
  id bigint generated by default as identity primary key,
  titulo varchar(255) not null,
  usuario varchar(255) null,
  senha text not null,
  descricao text null,
  criado_em timestamp without time zone default now(),
  atualizado_em timestamp without time zone default now()
);

-- Perfis e policies sugeridas para controle de acesso
-- Cria tabela de perfis ligada aos usuarios do auth
create table if not exists public.profiles (
  id uuid references auth.users(id) on delete cascade primary key,
  role varchar(20) not null default 'comum',
  created_at timestamp with time zone default now()
);

## SQL pronto: habilitar RLS, policies e exemplo de insert de perfil

-- Habilita RLS em todas as tabelas
alter table public.empresas enable row level security;
alter table public.links enable row level security;
alter table public.agenda_eventos enable row level security;
alter table public.senhas enable row level security;
alter table public.chamados enable row level security;
alter table public.profiles enable row level security;

-- Policies para `profiles` (cada usuário gerencia seu próprio perfil)
create policy "profiles_self_select" on public.profiles
  for select
  using (id = auth.uid());

-- Policy separada para INSERT
create policy "profiles_self_insert" on public.profiles
  for insert
  with check (id = auth.uid());

-- Policy separada para UPDATE
create policy "profiles_self_update" on public.profiles
  for update
  using (id = auth.uid())
  with check (id = auth.uid());

-- Policies para `empresas`: usuários autenticados podem ler todas
create policy "empresas_select_auth" on public.empresas
  for select
  using (auth.uid() is not null);

-- Policies para `links`: usuários autenticados podem ler todas
create policy "links_select_auth" on public.links
  for select
  using (auth.uid() is not null);

-- Se o frontend estiver recebendo 403 em links, rode também este bloco no SQL Editor:
alter table public.links enable row level security;
drop policy if exists "links_select_auth" on public.links;
create policy "links_select_auth" on public.links
  for select
  using (auth.uid() is not null);

-- 1. Garante que a API pode usar a área pública
grant usage on schema public to anon, authenticated;

-- 2. Dá permissão base para a API interagir com TODAS as tabelas
-- (Não se preocupe, o RLS ainda vai bloquear os dados sensíveis!)
grant all privileges on all tables in schema public to anon, authenticated;


-- Policies para `agenda_eventos`: usuários autenticados podem ler todas
create policy "agenda_eventos_select_auth" on public.agenda_eventos
  for select
  using (auth.uid() is not null);

-- Policy: apenas usuarios com role = 'chefe' acessam todas as rows de `senhas`
create policy "chefe_full_access_to_senhas" on public.senhas
  for all
  using (
    exists (
      select 1 from public.profiles p where p.id = auth.uid() and p.role = 'chefe'
    )
  )
  with check (
    exists (
      select 1 from public.profiles p where p.id = auth.uid() and p.role = 'chefe'
    )
  );

-- Opcional: permitir que usuarios autenticados criem chamados, mas somente 'chefe' pode deletar
create policy "auth_insert_chamados" on public.chamados
  for insert
  with check (auth.uid() is not null and created_by = auth.uid());

create policy "chefe_delete_chamados" on public.chamados
  for delete
  using (
    exists (
      select 1 from public.profiles p where p.id = auth.uid() and p.role = 'chefe'
    )
  );

-- Policy SELECT para chamados:
-- - 'chefe' pode ler todos os chamados
-- - 'comum' só pode ler os chamados que ele mesmo criou
create policy "chamados_select_chefe_all_comum_own" on public.chamados
  for select
  using (
    (exists (select 1 from public.profiles p where p.id = auth.uid() and p.role = 'chefe'))
    or
    (created_by = auth.uid())
  );

-- Policy UPDATE para chamados:
-- - 'chefe' pode editar todos os campos de qualquer chamado
-- - 'comum' só pode editar o status do chamado que ele criou
create policy "chamados_update_chefe_all" on public.chamados
  for update
  using (exists (select 1 from public.profiles p where p.id = auth.uid() and p.role = 'chefe'))
  with check (exists (select 1 from public.profiles p where p.id = auth.uid() and p.role = 'chefe'));

create policy "chamados_update_comum_own_status" on public.chamados
  for update
  using (created_by = auth.uid())
  with check (created_by = auth.uid() and (status is not null));
```

## Configuracao para deploy

### Backend

Defina estas variaveis de ambiente no backend:

- `DATABASE_URL`: connection string do Supabase.
- `DATABASE_SSLMODE=require`: obrigatorio para conexao externa.
- `CORS_ORIGINS`: URL publica do frontend na Vercel.
- `N8N_CHAT_WEBHOOK_URL`: opcional, se quiser manter a integracao do chatbot.

### Frontend na Vercel

Defina:

- `VITE_API_URL`: URL publica do backend.

## Observacoes importantes

- O backend usa `Base.metadata.create_all(...)` em [backend/database.py](backend/database.py), entao o schema acima precisa existir no Supabase antes do primeiro acesso em producao.
- Se quiser manter consistencia total entre app e banco, prefira criar as tabelas exatamente como acima e depois apontar o `DATABASE_URL` para o Supabase.
- O frontend consome apenas os endpoints de API, entao ele nao acessa o Supabase direto.
